## Inputs

## The style rubric (authoritative — follow verbatim)

For every row, produce a JSON object with this exact shape:

```json
{
  "request_id": "<copied verbatim from input>",
  "style_descriptors": {
    "formality": <float 0..1>,
    "verbosity": <float 0..1>,
    "warmth": <float 0..1>,
    "hedging": <float 0..1>,
    "humor": <float 0..1>,
    "directness": <float 0..1>
  },
  "distinctive_ngrams": ["<phrase>", "<phrase>", "<phrase>"],
  "style_card_md": "## Voice\n- <bullet>\n\n## Tells\n- <bullet>"
}
```

### Axis definitions (score each on a continuous [0..1] scale; 0.5 is neutral)

- **formality** — 0 casual ("gonna", slang, contractions, lowercase-start) / 1 formal (full sentences, no contractions, Latinate vocab, professional register).
- **verbosity** — 0 terse (short sentences, tight bullets) / 1 expansive (long winding sentences, qualifiers, subordinate clauses).
- **warmth** — 0 cold/detached (passive voice, no first/second person, no affect) / 1 warm (direct address, empathy markers, "you / we", explicit affect words).
- **hedging** — 0 assertive ("X is Y", no qualifiers) / 1 hedged ("I think", "perhaps", "somewhat", "it seems").
- **humor** — 0 dry/serious (no wordplay, no lightness) / 1 playful (puns, asides, self-deprecation, intentional irony, emoji).
- **directness** — 0 circuitous (preamble, throat-clearing, passive constructions) / 1 direct (imperative mood, no hedging, lead with the point).

Score honestly. The axes are correlated but not redundant — formal writing can still be warm; terse writing can still hedge. Use the middle of the range when a signal is mixed; reserve 0.0/1.0 for clear extremes.

### distinctive_ngrams

3–5 phrases (1–5 words each) that feel characteristically this writer's voice — NOT generic function words. Prefer n-grams that a log-odds-against-background scorer would flag as distinctive: content words, idioms, structural tics ("the thing is", "to be fair"). Each entry is at most 30 characters. If the text is too short or too generic to yield 3 distinctive n-grams, emit only what you can defend; never pad with filler.

### style_card_md

Short markdown, at most ~400 characters total. Two sections — `## Voice` and `## Tells` — each with 1–3 terse bullets. Voice bullets describe the overall register ("short declarative sentences, warm second-person"). Tells bullets call out specific giveaways ("uses 'fundamentally' as an intensifier", "ends imperatives with dashes"). Keep the card compact — this is a card, not an essay.

## Protocol

Execute these steps in order. Emit a short `agent.message` before each major step so the caller can follow progress on the Console timeline.

1. `bash`: capture the run start time: `date -u +%Y-%m-%dT%H:%M:%SZ > /workspace/started_at.txt`.

2. `read` `/mnt/session/uploads/workspace/input.jsonl`. Parse the rows.

3. Score rows in batches of ~50. For each batch:
   - Produce the full style-profile JSON object for each row in the batch (copy `request_id` through verbatim).
   - `write` the batch as JSONL to `/workspace/style-profiles.jsonl` in append mode (use `bash` `cat >> …` with a heredoc, because `write` may overwrite).
   - Emit an `agent.message` progress update of the form:
     ```
     PROGRESS {"profiles_done": N, "profiles_total": T, "batch": K}
     ```
     where `N` is rows labeled so far, `T` is total input rows, `K` is the batch index (1-based).

4. After all rows are scored, compute and `bash write` `/workspace/run_manifest.json`:
   ```json
   {
     "component": "style-extractor",
     "run_type": "pilot",
     "opus_model": "claude-opus-4-7",
     "started_at": "<from /workspace/started_at.txt>",
     "finished_at": "<current ISO UTC>",
     "input_count": <number of input rows>,
     "profiles_written": <number of output rows>,
     "skipped_count": <0 unless you had to skip a row — explain why in the summary if you did>,
     "axis_means": {
       "formality": <mean across all profiles>,
       "verbosity": <mean>,
       "warmth": <mean>,
       "hedging": <mean>,
       "humor": <mean>,
       "directness": <mean>
     },
     "aborted": false
   }
   ```

5. Final message — emit ONE `agent.message` that contains, in this exact order:
   - A single line: `RUN_MANIFEST_BEGIN`
   - The full contents of `/workspace/run_manifest.json` (pretty-printed JSON, multi-line ok).
   - A single line: `RUN_MANIFEST_END`
   - A single line: `STYLE_PROFILES_BEGIN`
   - The full contents of `/workspace/style-profiles.jsonl` (one JSON object per line, exactly as written).
   - A single line: `STYLE_PROFILES_END`
   - A single line: `RUN_COMPLETE`

   The caller's monitor script parses these markers to extract the manifest and profiles. No other text between the markers.

## Hard stops (abort cleanly, do not emit `RUN_COMPLETE`)

- Cumulative wall-clock > 60 minutes since `started_at`.
- More than 15 rows you could not profile confidently even after a fresh attempt (text genuinely un-profileable — pure log output, non-English content beyond your reliable coverage, etc.).

On a hard stop:
- `bash write` `/workspace/run_manifest.json` with `"aborted": true` and a short `"reason"` string.
- Emit ONE `agent.message` with `RUN_ABORTED` on a line, followed by the aborted manifest between `RUN_MANIFEST_BEGIN` / `RUN_MANIFEST_END` markers.
- Stop.

## Output discipline — read this twice

The style_card_md MUST stay under ~400 characters and each n-gram under 30 chars. This is a hard budget constraint: the pilot budget math assumes ~200 output tokens per row. If you let cards sprawl to essays, the run blows the budget. Keep cards dense and evidence-tight.

## Tone for progress messages

Terse — a single line or two. The Console timeline is your log; the final `RUN_COMPLETE` message is your delivery vehicle. Do not narrate your internal reasoning to the user; the `agent.thinking` channel handles that if extended thinking is enabled.
